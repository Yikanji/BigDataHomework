## ANNS 在你任务里的位置

你 Paper 2 (Retrieval-Augmented VLA / Cross-Embodiment Memory) 的核心流程:

```
当前观测 + 指令 + 本体状态
        ↓
  [编码成 query embedding]      ← Encoder(你要训)
        ↓
  [从百万级 episode 库检索 top-K]  ← ANNS(这里用 PathWeaver/FusionANNS 这类)
        ↓
  [把检索到的 episode 注入 VLA]   ← 注入机制(你要设计)
        ↓
       动作输出
```

**ANNS 就是中间那一步**——给你一个 query 向量,从百万级向量库里快速找出最相似的 K 个。

## 一个具体例子

假设你机器人要执行"把红色易拉罐递给我"这个任务:

1. **当前观测** = RGB 图像(看到桌面有多个物品) + 指令文本 + 关节状态
2. **编码** = 通过一个 encoder 把这些信息压成一个 1024 维向量 `q`
3. **ANNS 检索** = 从你之前录的 100 万条 episode 库里,找出 5 个最相似的历史经验
   - 这一步就是 ANNS 的工作:在 100 万个 1024 维向量里找 top-5
   - **要求**:延迟 < 20ms(因为 50ms 控制周期里还要给 VLA 推理留时间)
4. **注入 + 推理** = 把这 5 个相似经验作为 in-context 喂给 VLA,生成下一步动作

**没有 ANNS,第 3 步就是不可能完成的任务**——你不可能每次控制周期都遍历 100 万条 episode 算距离。

## Paper 2 是什么:Cross-Embodiment Memory

**一句话**:让机器人在执行任务时,从一个**跨本体**的 episode 库里检索相关历史经验,辅助 VLA 决策。

**核心 idea**:
- Memory 里存所有来源的 episode:OXE + AgiBot World + 你自采(xArm/G1/各种本体混杂)
- 检索时**不要求本体相同**——G1 可以检索 Franka 的成功经验来用
- Pons(三层架构的中间层)负责把"别人本体的经验"翻译成"自己本体能执行的"context tokens

**关键技术挑战**:
1. **Action representation alignment**:xHand 16 维"夹紧" vs G1 单维"夹紧"——怎么对齐语义
2. **多模态低延迟检索**:50ms 控制周期内完成,这是 ANNS 系统层的硬骨头
3. **Quality-aware retrieval**:检到失败 episode 会反噬,要做正负样本区分
4. **Multi-vector indexing**:一个 episode 不是单向量(几百帧轨迹)

**双 paper 策略**:
- ML 顶会版 (NeurIPS/CoRL):action alignment + cross-embodiment generalization
- DB 顶会版 (VLDB/SIGMOD):多模态低延迟检索引擎 + episode 索引
- **一份代码,两个故事**

**为什么你做最合适**:你有 G1 + xArm + xHand 多本体真机,跨本体实验只有你能做扎实——别人不可复制。

## 具身智能需要数据系统的 7 个具体场景

### 1. 训练数据的组织和筛选

**现在的痛点**:

**对比传统场景**:互联网公司的数据仓库(Snowflake、BigQuery)就是为这种"按多维度筛选 + 聚合"设计的,但它们处理不了视频、轨迹、张量。

**对应你的工作**:Paper 2 必须解决"按 episode 属性 + 多模态相似度联合过滤"的查询。

---

### 2. 训练时的高效采样

**现在的痛点**:
- VLA 训练要按 time window、按下采样率、按事件触发条件抽片段
- Replay buffer 100GB+,优先级采样靠内存 sum-tree,scale 不上去
- 多 GPU 训练时,数据 loader 经常是瓶颈

**对比传统场景**:数据库的索引和物化视图(materialized view)正是解决"高频查询的不同切面"——TSDB 的 continuous aggregate 接近这个需求,但不支持轨迹语义。

---

### 3. 推理时的实时检索(就是 Paper 2 的核心)

**现在的痛点**:
- 控制周期 50ms,VLA forward 10-30ms,留给 retrieval **< 20ms**
- 库规模:百万级 episode × 多模态(视觉 + 轨迹 + 语言)
- 查询是复合的:"找视觉相似 + 任务相似 + 同本体 + 成功的 episode"
- 现有方案:Faiss 不带元数据过滤、Milvus 延迟波动大、纯 ANNS 不懂任务语义

**这是数据库的核心战场**,不是 ML 框架能解决的。

---

### 4. 真机部署时的遥测和回放

**现在的痛点**:
- 机器人 deploy 时持续产生数据:视觉 30Hz、proprio 1kHz、tactile 1kHz、力矩 1kHz、动作命令、VLA 中间状态

**对比传统场景**:可观测性数据库(Prometheus、Honeycomb)就是为"实时遥测 + 事后分析"设计的,但它们假设标量监控指标,处理不了张量+视频。

---

### 5. Sim-to-Real 的数据血缘

**现在的痛点**:

### 6. 跨本体数据对齐

**现在的痛点**(这是你硬件优势对应的痛点):

- 每个 lab 自己写一套 adapter,代码不可复用

**对比传统场景**:这就像 21 世纪初没有 ORM,每个公司自己写一遍 SQL → Object 映射——后来 Hibernate / SQLAlchemy 解决了,因为有数据系统统一抽象。

### 7. 持续学习的数据演化

**现在的痛点**:

**对比传统场景**:OLTP 数据库的事务和索引维护在 30 年前就解决了,但向量库的在线维护还很初级。

## 向量库的 7 个盲区(对应你的具体需求)

### 盲区 1:Episode ≠ 单个向量

**向量库的假设**:1 entity = 1 vector。

**你的现实**:一个 episode 是 200-2000 帧的时序,包含:
- 视觉轨迹(每帧一个 embedding)
- 动作轨迹(每帧一个 action vector)
- proprioception 轨迹
- 一个语言指令
- 元数据(成功标志、本体、任务类别)

塞进向量库的常见 hack:
- **方案 A**:整个 episode 池化成单向量 → 丢掉时序结构,粒度太粗
- **方案 B**:每帧一个向量,episode_id 当 metadata → 1M episode × 200 帧 = 2 亿向量,查询时还要 group by episode
- **方案 C**:ColBERT-style late interaction → 学术界在搞,**没有任何主流向量库原生支持**

**这就是 Paper 2 一个核心 contribution 空间**:multi-vector index for episodic data。

### 盲区 2:相似度函数对不上

**向量库**:余弦距离、欧氏距离、内积——固定几个。

**你的需求**:任务相似度 ≠ 视觉相似度 ≠ 余弦相似度。
- 两张几乎一样的桌面图,物体重量不同 → 视觉余弦距离 = 0.99,但需要的策略完全不同
- 两段轨迹空间形状不同,但都是"绕开障碍抓取" → 应该相似但欧氏距离很远

你需要的是 **task-aware embedding + learned similarity**。向量库只管索引,不管 embedding 怎么训——这部分是你的研究 substance。

### 盲区 3:多模态联合查询

**向量库**:一次查询 = 一个向量。要做多模态要么 concatenate(粗暴)要么 weighted sum(参数怎么定)。

**你的需求**:查询是 (视觉, 语言指令, 当前 proprioception) 联合。"视觉相似 0.8 + 语言相似 0.9 + 本体状态相似 0.6"怎么合成一个 ranking?

主流向量库支持 hybrid search(BM25 + 向量)是 2 路融合,**3 路以上多模态融合没有标准方案**。

### 盲区 4:延迟波动 vs 硬实时

**向量库**:目标是 p99 < 100ms,不保证 worst case。

**你的需求**:50ms 控制周期,**任何一次超过 20ms 就是控制失败**——机器人卡顿、抖动、甚至失控。

Milvus 实测 p99 可能 30ms,但 p99.9 可能突然飙到 200ms(GC、index rebuild、节点抖动)。这对 web search 无所谓,对机器人控制是灾难。

**硬实时 SLA 是数据库会议正经的 systems contribution**——Paper 2 DB 版的核心卖点之一。

### 盲区 5:元数据过滤不够强

**向量库**:支持 metadata filter,但都是简单的 `tag = "value"` 这种。

**你的需求**:
- "成功率 > 80% **且** 同本体 **且** 任务类别 ∈ {抓取, 递交} **且** 时长 < 30 秒 **且** 不要含力矩异常"
- 这种复合谓词在向量库里要么走 pre-filter(召回率掉)、要么走 post-filter(性能掉)

**Predicated ANN search 在 DB 顶会上还在活跃研究**(2024 SIGMOD 有几篇),你的场景是这个研究的下游应用,可以直接消费。

### 盲区 6:在线插入和演化

**向量库**:大多假设静态库 + 周期性 reindex。少数支持在线插入但性能下降明显。

**你的需求**:机器人 deploy 时持续产生新 episode(每天几百到几千条),memory 要在线增长 **且**保持检索性能。

更狠的:有些 episode 跑着跑着发现是失败的(几小时后才确认),需要 **retroactive update**——这是事务数据库的强项,向量库基本不支持。

### 盲区 7:无任务语义层

**向量库**:就是个 ANN engine,完全不懂"episode"、"任务"、"动作"、"成功"。

**你的需求**:一整套 episode 语义层——episode 边界管理、跨 episode 关系、任务标签体系、动作 schema 跨本体对齐。

**这不是向量库的职责,但目前没人做**——这就是你 Paper 2 的系统空间。

## 那这些向量库到底能干什么

**能干的事**:作为你系统的**底层索引引擎**(plumbing),不是顶层接口。

具体落地:
- **Phase 0 prototype 阶段**:Faiss 直接用,百万向量内单机能跑,验证 retrieval-augmented VLA 的算法假设
- **Paper 1 阶段**:Faiss / Qdrant 当 baseline,你 Paper 2 DB 版要 beat 的对象
- **Paper 2 阶段**:在 Faiss / DiskANN 基础上,**加你的 episode 层、跨本体层、硬实时层**

类比:就像建一个 web app,你用 PostgreSQL 当底层引擎,但要在上面加 ORM、business logic、API 层——**没人会说"我直接用 PostgreSQL 就能解决所有问题"**,但也没人重新造数据库。

## 先解决你的根本问题:NN vs ANN

### 精确最近邻 (Exact NN)

给你一个 query 向量 `q`,从 N 个向量 `{v_1, ..., v_N}` 里找出和 `q` 最相似的 K 个。

**朴素做法**:遍历所有 N 个向量,逐个算距离,排序取前 K。
- 时间复杂度 O(N × d),d 是向量维度
- N = 1M, d = 1024 时,**单次查询 100ms+**——对你 50ms 控制周期完全不可用

### 近似最近邻 (ANN / ANNS)

**核心思想**:**牺牲一点点准确率(找到的不是真正 top-K,而是"差不多 top-K"),换取数量级的速度提升**。

实际效果:
- 1M 向量,d=1024
- 精确 NN:100ms,100% recall
- ANNS:**1-5ms**,95% recall(意思是 top-10 里平均有 9.5 个是真正的 top-10)

**这个 trade-off 在你的场景完全可以接受**——VLA retrieval 不需要数学意义上的最优,"相关 episode 在 top-K 里就够"。

## ANNS 的三大算法家族

不需要会写,但你要知道差别——选型时关键。

### 1. 基于图 (Graph-based) ⭐ 最常用

**代表**:HNSW、NSG、DiskANN、CAGRA

**核心思想**:把所有向量构建成一个图,相似的向量互为邻居。查询时从某个起点出发,贪心地往"离 query 更近的邻居"走,直到无法更近。

```
      v_3 ─── v_7
     / │       │ \
   v_1 │       │  v_9
     \ v_5 ── v_8 /
      \  │     │ /
       \ │     │/
        query 从某点起,贪心找最近邻
```

**优点**:Recall 高、延迟稳定、SOTA 性能
**缺点**:构图慢、内存占用大、不太好分布式

**你应该用哪个**:
- 内存够 → **HNSW** (最成熟,Faiss/Milvus/Qdrant 都内置)
- 内存不够要走 SSD → **DiskANN** (微软的,百亿级别)
- GPU 加速 → **CAGRA** (NVIDIA RAPIDS,你之前看的 PathWeaver 就改进这个)

### 2. 基于量化 (Quantization-based)

**代表**:PQ (Product Quantization)、IVF-PQ、OPQ

**核心思想**:把高维向量"压缩"成几个字节的 code,距离计算用 code 近似。

- 原始:1024 维 × float32 = 4KB/向量
- PQ 压缩后:64 字节/向量 = **压缩 64 倍**
- 距离计算变成查表,极快

**优点**:超省内存、超快
**缺点**:Recall 比图方法低一点
**典型用法**:**和图方法组合**——图导航 + PQ 算距离,这是 SOTA(比如 FusionANNS 就用了 PQ + 多层索引)

### 3. 基于哈希 (Hash-based)

**代表**:LSH (Locality-Sensitive Hashing)

**核心思想**:相似的向量哈希到相同桶里。
**现状**:学术界经典方案,**工业界基本不用了**——被图和量化方法吊打

## 具体怎么用 ANNS

### Phase 0(原型阶段)

**直接用 Faiss**:
```python
import faiss

# 建索引
index = faiss.IndexHNSWFlat(1024, 32)  # 1024 维,每节点 32 邻居
index.add(episode_embeddings)  # 百万级向量灌进去

# 查询
D, I = index.search(query, k=50)  # 5ms 拿到 top-50
```

10 行代码,百万级别,单机 5ms。Paper 1 阶段够用。

### Paper 2(系统贡献阶段)

这时候 Faiss 不够,你需要解决:

**1. 多模态联合 ANNS**(向量库的盲区 3)
- 视觉向量 + 语言向量 + proprio 向量,不是一个 query 而是三个
- 三路 ANNS 怎么并行 + fusion?Reviewer 没见过的问题

**2. 谓词过滤 ANNS**(向量库的盲区 5)
- "成功率 > 80% + 同本体 + 任务 ∈ 抓取" 等元数据过滤
- pre-filter 召回掉,post-filter 性能掉,**hybrid 路由是研究问题**

**3. 硬实时 ANNS**(向量库的盲区 4)
- p99.9 < 20ms,worst-case 必须保证
- 现有 ANNS 都是 p99 SLA,**硬实时 ANNS 是新领域**

**4. 流式 ANNS**(向量库的盲区 6)
- 新 episode 持续插入,索引不能 rebuild
- HNSW 在线插入会逐渐退化,**动态索引维护是开放问题**

**这四条每一条都能成 Paper 2 系统部分的核心 contribution**——而且都和你之前看的两篇 ANNS paper (PathWeaver、FusionANNS) 是同一个 community 的工作,reviewer 一眼能看懂你在解决什么问题。

## ANNS 和你看的两篇 paper 重新串起来

现在你回头看王旭然汇报的两个工作,就明白它们在干嘛了:

### PathWeaver (USENIX ATC'25)
- 解决**图方法在多 GPU 上的扩展性**
- 关键洞察:多 GPU 不要简单分片,要做流水线
- 它优化的是 ANNS 算法本身的执行效率

### FusionANNS (FAST'25)
- 解决**十亿级 ANNS 的 CPU/GPU/SSD 协同**
- 关键洞察:不同硬件层放不同表示(原始向量在 SSD,压缩向量在 GPU,索引在 CPU 内存)
- 它优化的是 ANNS 系统的 IO 和异构计算

**这两个工作和你的关系**:
- 它们做"让单一 ANNS query 更快"
- 你做"让 ANNS query 支持具身场景的特殊需求"
- **互补,不竞争**

实际 Paper 2 实现时,你**底层会调用类似 FusionANNS 这种 SOTA ANNS engine**,在它上面加你的多模态、谓词、硬实时、流式四层创新。
